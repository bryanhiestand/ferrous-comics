## Bugs and Correctness Issues

### 1. `backfill_limit = 0` still sends `latest.num` — this is a behavior change, not a no-op

**Location:** `main()`, lines 485-489


The documentation says `0 = disabled`. But the `else` branch always pushes `latest.num` into candidates regardless of whether it has already been seen. The `is_seen` check at the top of the loop catches it and skips it, so no duplicate email is sent — but the semantics are still wrong: a user who sets `backfill_limit=0` expecting "no backfill, just normal behavior" actually gets exactly normal behavior, so functionally it works. The issue is the opposite case: a user who sets `backfill_limit=0` expecting to **disable all comic sending** (a reasonable reading of "disabled") will still get the latest comic emailed. The variable name and its documented meaning are ambiguous. This is at minimum a documentation/naming bug, and could be a real behavior bug depending on operator intent.

### 2. `last` overflow: `last.map(|m| m + 1)` can panic on `u32::MAX`

**Location:** `main()`, line 484


If the highest recorded comic number is `u32::MAX` (4,294,967,295), this overflows and panics in debug builds, or wraps to 0 in release builds (producing `0..=latest.num`, a huge candidate list). xkcd is at comic ~3100 today, so this will not occur in practice for decades, but Rust's default integer arithmetic does not make it safe by construction. ### 3. `fetch_comic_by_num` skips on any error, including transient network failures — this silently drops comics permanently

When a comic is skipped due to a network timeout or a 5xx error, the loop continues to the next comic. On the next cron run, `last_seen_num` will return the last *successfully written* comic, so skipped comics will be retried (the database was not updated for them). This is correct behavior.

To fix: after a 404 response in `fetch_comic_by_num`, the caller should mark the comic as seen (possibly with a special flag) to advance past it, rather than treating it identically to a transient transport error.
